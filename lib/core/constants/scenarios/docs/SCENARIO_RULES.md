# 시나리오 작성 규칙

시나리오 스크립트(ScenarioScript) 작성 시 반드시 따라야 하는 규칙.
비용 레퍼런스: `lib/core/constants/starcraft_costs.dart`

---

## 1. 양쪽 동시 비용 반영 (핵심 원칙)

**모든 이벤트에서 home과 away 양쪽의 자원/병력 변동을 동시에 반영한다.**

스타크래프트에서 양쪽 선수는 동시에 건물을 짓고 유닛을 생산한다.
한쪽 행동만 기록하면 상대는 recovery(채광)만 반영되어 부자연스러운 자원 격차가 생긨.

### 미러 매치업

양쪽이 같은 빌드 → 하나의 이벤트에 양쪽 동일 비용.

```dart
// AS-IS (잘못된 예시) - 홈만 비용 반영
ScriptEvent(
  text: '{home} 선수 배럭 건설!',
  owner: LogOwner.home,
  homeResource: -150,
  fixedCost: true,
),
ScriptEvent(
  text: '{away} 선수도 배럭 건설!',
  owner: LogOwner.away,
  awayResource: -150,
  fixedCost: true,
),

// TO-BE (올바른 예시) - 양쪽 동시 반영
ScriptEvent(
  text: '{home} 선수 배럭 건설! {away} 선수도 배럭!',
  owner: LogOwner.system,
  homeResource: -150,
  awayResource: -150,
  fixedCost: true,
  altText: '양쪽 배럭이 올라갑니다!',
),
```

### 크로스 매치업

양쪽이 다른 빌드 → 각 이벤트에 상대의 동시 행동 비용도 포함.

```dart
// BBS(home) vs 노배럭더블(away)
// 홈이 배럭 2개 지을 때, 어웨이는 커맨드센터 건설 중
ScriptEvent(
  text: '{home} 선수 센터 배럭! 본진에도 배럭!',
  owner: LogOwner.home,
  homeResource: -300,  // 배럭 x2
  awayResource: -400,  // CC (상대의 동시 행동)
  fixedCost: true,
  altText: '{home}, BBS! {away} 선수는 앞마당 확장!',
),
```

### 전투 이벤트

전투 중에도 양쪽은 후방에서 유닛을 생산하고 있다.
전투 이벤트에서 직접적인 전투 피해 외에도 양쪽 생산 비용을 포함할 수 있다.

```dart
// 전투 중 양쪽 탱크 추가 생산
ScriptEvent(
  text: '{home} 선수 시즈 포격! 상대 탱크를 직격!',
  owner: LogOwner.home,
  awayArmy: -4,       // 전투 피해
  homeArmy: -2,       // 맞교환
  homeResource: -250,  // 홈 탱크 추가 생산 (후방)
  awayResource: -250,  // 어웨이도 탱크 생산 (후방)
  favorsStat: 'attack',
),
```

## 2. 비용 체계

### 자원 (Resource)

- 미네랄 + 가스 합산 단일값
- 시작값: 50 (실제 SC 시작 미네랄)
- 범위: 0 ~ 10,000
- 0 미만은 자동 클램핑 (비용이 자원보다 크면 0으로)

### 병력 (Army)

- 서플라이 기반 (마린 1, 탱크 2, 캐리어 6)
- 시작값: 0
- 범위: 0 ~ 200
- 일꾼(SCV/프로브/드론)은 병력에 미포함

### fixedCost

- 건물 건설, 유닛 생산, 업그레이드 연구 → `fixedCost: true`
- 전투 피해, 견제 피해 → `fixedCost: false` (기본값)
- fixedCost: true인 이벤트는 favorsStat/winRate 모디파이어가 적용되지 않음
- fixedCost: false인 이벤트만 능력치 차이에 의한 보정이 적용됨

### 비용 분할 원칙 (핵심)

**큰 비용은 개별 건물/유닛 단위로 분할한다.**

자원은 recovery를 통해 점차 쌓인다. 한 이벤트에 여러 건물을 묶으면:
- 클램핑(0 미만 → 0)으로 실제 비용이 흡수됨
- 표시되는 자원 변동이 실제 비용과 괴리됨

```dart
// AS-IS (잘못된 예시) - 배럭 2개를 한 이벤트에
ScriptEvent(
  text: '{home} 센터 배럭! 본진 배럭! {away}도!',
  homeResource: -300, // 배럭 x2
  awayResource: -300,
  fixedCost: true,
),

// TO-BE (올바른 예시) - 건물 하나씩 분할
ScriptEvent(
  text: '{home} 선수 센터에 배럭! {away} 선수도 센터 배럭!',
  homeResource: -150, // 센터 배럭
  awayResource: -150,
  fixedCost: true,
),
ScriptEvent(
  text: '{home} 선수 본진에 배럭! {away} 선수도 본진 배럭!',
  homeResource: -150, // 본진 배럭
  awayResource: -150,
  fixedCost: true,
),
```

**분할 기준:**
- 건물은 개별 건물 단위로 분할 (배럭 150, 팩토리 300, 스타포트 250 등)
- 리파이너리/가스는 별도 이벤트 (100)
- 유닛 생산은 같은 종류끼리 묶기 가능 (마린 3기 = -150)
- 업그레이드 연구는 별도 이벤트 (시즈모드 300, 클로킹 300 등)
- 건물 + 업그레이드를 한 이벤트에 묶지 않는다

**자원 흐름 예시 (분할 후):**
```
Start: 50
L1: +100=150 → SCV 정찰(비용 없음) → 150
L2: 150 → 센터 배럭(-150) → 0 → +100 = 100  (delta: -50)
L3: 100 → +100 = 200 → 본진 배럭(-150) → 50 → +100 = 150  (delta: -50... 실제로는 event 먼저 적용)
```

> **엔진 처리 순서**: `이전 state → 이벤트 비용(clamp 0) → +recovery → 최종 state`
> 표시 delta = `비용 + recovery` (예: -150 + 100 = -50). 채광하면서 건설하는 실제 SC와 동일.

## 3. Recovery (채광 수입)

매 줄(이벤트 줄 포함)마다 양쪽 모두에게 자동 적용.
서플라이디팟/파일런/오버로드 비용은 recovery에 암묵적으로 포함.

| 게임 단계 | startLine 기준 | recoveryResourcePerLine | recoveryArmyPerLine |
|-----------|---------------|------------------------|---------------------|
| 오프닝    | 1~15          | 100                    | 0                   |
| 얼리미드  | 16~35         | 150                    | 1                   |
| 미드게임  | 36~60         | 200                    | 2                   |
| 레이트    | 60+           | 300                    | 3                   |

### 확장 보너스

앞마당/추가 확장 시 `homeExpansion: true` 또는 `awayExpansion: true` 플래그를 설정하면
해당 측의 recovery가 **확장당 +75** 증가한다.

```dart
// 홈이 앞마당 확장 (커맨드센터/넥서스/해처리)
ScriptEvent(
  text: '{home} 선수 앞마당에 커맨드센터를 건설합니다!',
  owner: LogOwner.home,
  homeResource: -400, // 커맨드센터 400
  fixedCost: true,
  homeExpansion: true, // ← 홈 recovery +75 증가
),
```

| 확장 수 | 오프닝 recovery | 미드게임 recovery |
|---------|----------------|-----------------|
| 본진만  | 100            | 200             |
| 1확장   | 175            | 275             |
| 2확장   | 250            | 350             |

### 프로 자원관리 감쇠

프로 선수는 돈을 남기지 않는다. 자원이 400 이상이면 recovery가 자동 감쇠:

| 자원 | recovery 비율 | 예시 (base 100) |
|------|-------------|----------------|
| 0~400 | 100% | 100 |
| 600 | 75% | 75 |
| 800 | 50% | 50 |
| 1000+ | 25% (하한) | 25 |

→ **자원이 500~1000 범위에서 자연 유지**됨. 건설/생산으로 자원을 쓰면 recovery가 다시 올라감.

### 자원 흐름 예시 (BBS 미러, recovery=100)

```
엔진 처리 순서: 이전 state → 이벤트 비용(clamp 0) → +recovery → 최종 state

L1: 50 → +100 = 150  SCV 정찰 (비용 없음)
L2: 150 - 150(센터 배럭) = 0 → +100 = 100
L3: 100 - 150(본진 배럭) = clamp(0) → +100 = 100
L4: 100 - 150(마린 3기, +3army) = clamp(0) → +100 = 100
...Phase 1 시작 전 빈 줄에서 자원 축적...
L6: 577 - 100(마린+SCV 전진) → +89(감쇠) = 577
L8: 636 - 100(벙커) → +73(감쇠) = 636
```

## 4. 이벤트 구조 규칙

### owner 기준

| owner | 사용 시점 | 예시 |
|-------|----------|------|
| `LogOwner.system` | 양쪽 동시 행동, 해설 코멘터리 | 건물 동시 건설, 상황 설명 |
| `LogOwner.home` | 홈 선수가 주도하는 행동 | 홈 공격, 홈 컨트롤 |
| `LogOwner.away` | 어웨이 선수가 주도하는 행동 | 어웨이 수비, 어웨이 반격 |

### 오프닝 이벤트 (빌드업)

- 양쪽 건물/유닛 생산은 가능하면 하나의 이벤트로 합침
- owner: `LogOwner.system` (양쪽 동시)
- `fixedCost: true` 필수
- 양쪽 비용을 모두 명시

```dart
ScriptEvent(
  text: '{home} 선수 팩토리! {away} 선수도 팩토리 건설!',
  owner: LogOwner.system,
  homeResource: -300,
  awayResource: -300,
  fixedCost: true,
),
```

### 크로스 매치업에서 동시 행동 비용 산정

한쪽 이벤트를 서술할 때, 상대가 **같은 시간에 하고 있을 행동**의 비용을 추가.

| 홈 이벤트 | 어웨이 동시 행동 | awayResource |
|-----------|-----------------|--------------|
| 배럭 건설 | CC 건설 | -400 |
| 마린 생산 | 가스+팩토리 건설 | -400 |
| 벌처 돌진 | 탱크 생산 (후방) | -250 |
| 시즈 포격 | 탱크 추가 생산 | -250 |

**빌드오더 타임라인**을 기준으로 동시 행동을 결정.
실제 SC에서 해당 시점에 상대가 무엇을 하고 있을지를 고려.

### 전투 이벤트 (분기)

- 전투 피해: `fixedCost: false` (능력치 보정 적용)
- 전투 중 생산: `fixedCost: true`인 별도 이벤트 또는 같은 이벤트에 포함
- 전투 이벤트에서도 상대의 후방 생산을 반영 가능

### decisive 이벤트

- `decisive: true`인 이벤트는 즉시 승패 판정
- 결전 분기에서 home/away 각 1개씩 배치
- decisive 이벤트에는 자원/병력 변동 없이 텍스트만 (승패는 엔진이 판정)

## 5. Phase 구성

### 기본 구조

```
Phase 0: 오프닝 (startLine 1)     - 양쪽 빌드업 (건물+유닛)
Phase 1: 전개   (startLine ~15)   - 첫 교전, 견제
Phase 2: 분기   (startLine ~22)   - 공격 성공/실패 분기
Phase 3: 중반   (startLine ~30)   - 후속 전개
Phase 4: 결전   (startLine ~38+)  - decisive 분기 (home_wins / away_wins)
```

### linearEvents vs branches

- **오프닝/빌드업**: `linearEvents` (순차)
- **교전/결전**: `branches` (분기)
- 분기는 `conditionStat`(능력치 조건) + `baseProbability`(확률)로 선택

## 6. 텍스트 규칙

### 플레이스홀더

- `{home}`: 홈 선수 이름 (역방향 시 자동 스왑)
- `{away}`: 어웨이 선수 이름

### altText

- 50% 확률로 대체 텍스트 사용
- 텍스트 다양성 확보용

### skipChance

- 0.0 ~ 1.0, 해당 확률로 이벤트 스킵
- 시스템 해설 이벤트에 0.3~0.5 권장
- 필수 빌드업 이벤트에는 사용 금지

### 금지어

- '경제' → '자원', '일꾼', '확장' 사용
- '포토캐논' → '캐논'
- '쓰러지다' → '무너지다', '밀리다'
- '착공' → '건설합니다', '올립니다', '짓기 시작합니다' 등 (공사 용어, 해설 톤 아님)
- '경쟁' → '싸움' 사용 (예: '뮤탈 타이밍 싸움', '공중전 돌입'). 실제 해설에서 '경쟁'은 부자연스러움. 예외: '테크 경쟁' → '테크를 올립니다' 등 문맥에 맞게 변환
- 'X 미러' → 'X' 미러 직접 언급 금지 (예: '레어 미러!' → '동일한 빌드!'). 실제 해설에서 미러라는 표현을 쓰지 않음
- 건물명은 CLAUDE.md 표기 기준 준수

## 7. 역방향 매칭 (isReversed)

> `homeBuildIds`/`awayBuildIds` 겹침 금지 원칙은 CLAUDE.md "Home/Away 규칙" 참조.

역방향 매칭 시 엔진이 자동으로 스왑:
- homeArmy ↔ awayArmy
- homeResource ↔ awayResource
- {home} ↔ {away} 텍스트
- LogOwner.home ↔ LogOwner.away

**양쪽 동시 비용 이벤트는 역방향에서도 자연스럽게 대칭됨.**

## 8. 미러전 확장 타이밍 분기 (필수)

미러전에서 확장(앞마당/트리플)은 **항상 동시에 진행하면 안 된다.**
확장 타이밍을 분기(branch phase)로 구현하여 다양한 게임 전개를 만든다.

### 분기 유형 (5가지)

| 분기 | baseProbability | 설명 | decisive |
|------|----------------|------|:--------:|
| **동시 확장** | 1.5 | 양쪽 비슷한 타이밍에 확장. 가장 빈번 | X |
| **홈 빠른 확장** | 0.7 | 홈이 먼저 확장 → 어웨이가 병력 우위로 압박. 버티면 홈 자원 우위 | X |
| **어웨이 빠른 확장** | 0.7 | 미러 | X |
| **홈 확장 스킵 공격** | 0.4~0.6 | 홈이 확장 포기, 병력 올인 → 상대 확장 타이밍에 밀어붙임 | O |
| **어웨이 확장 스킵 공격** | 0.4~0.6 | 미러 | O |

### 구현 패턴

```dart
// 확장 타이밍 분기 phase (트리플 확장 예시)
ScriptPhase(
  name: 'triple_expansion',
  startLine: 58,
  recoveryArmyPerLine: 2,
  recoveryResourcePerLine: 200,
  branches: [
    // 동시 확장 (가장 빈번)
    ScriptBranch(
      id: 'both_expand',
      baseProbability: 1.5,
      events: [
        ScriptEvent(text: '{home} 선수 세 번째 확장을 올립니다.', ...),
        ScriptEvent(text: '{away} 선수도 세 번째 확장을 올립니다.', ...),
        // 벌처 견제 이벤트 (skipChance 포함)
      ],
    ),
    // 빠른 확장 (리스크, decisive 아님)
    ScriptBranch(
      id: 'home_fast_expand',
      baseProbability: 0.7,
      events: [
        // 홈 확장 → 어웨이 병력으로 압박 → 홈 버티기
        // 확장 비용으로 병력이 얇아지는 내러티브
      ],
    ),
    // 확장 스킵 공격 (올인, decisive)
    ScriptBranch(
      id: 'home_skip_expand_attack',
      baseProbability: 0.4,
      conditionStat: 'attack',
      events: [
        // 홈 확장 포기 → 병력 추가 생산 → 상대 확장 타이밍에 공격
        // 마지막 이벤트에 decisive: true
      ],
    ),
  ],
)
```

### 확장 선택의 결과 반영

확장 분기의 결과는 이후 phase에 자연스럽게 반영되어야 한다:
- **빠른 확장 성공**: 자원 우위 → 이후 더 많은 병력/테크 가능
- **빠른 확장 실패**: 병력 열세 → 상대에게 밀림
- **확장 스킵 공격**: decisive로 게임 종료 또는, 실패 시 자원 열세로 장기전 불리

### 빌드별 확장 시점

| 빌드 | 앞마당 (2번째) | 트리플 (3번째) | 비고 |
|------|-------------|-------------|------|
| 배럭더블 | 오프닝 (빌드 포함) | Phase 4~5 (분기) | 앞마당은 빌드 일부 |
| 팩토리더블 | 오프닝 (빌드 포함) | Phase 4~5 (분기) | 앞마당은 빌드 일부 |
| 투팩 벌처 | Phase 3 (분기) | Phase 6~7 (분기) | 앞마당이 늦음 |

## 9. 중반 종료 분기 (탱크 푸시 타이밍)

벌처전 이후 탱크 전환 직후, 병력 우위 쪽이 시즈 모드 완료 전에 밀어붙이는 조기 종료 분기.
29~42줄(중반) 구간에서 게임이 끝날 수 있도록 한다.

```dart
// 탱크 전환 직후 밀어붙이기 체크
ScriptPhase(

  startLine: 46,  // tank_vulture 직후
  branches: [
```

## 10. 드랍 운영 다양성

TvT 후반 드랍십은 항상 올인이 아니다. 3가지 유형으로 분류한다.

### 드랍 유형

| 유형 | baseProbability | 설명 | decisive |
|------|----------------|------|:--------:|
| **게릴라 드랍** | 1.0~1.2 | 확장 견제 후 회수. 자잘한 피해 축적 | X |
| **마무리 드랍** | 0.6~0.7 | 유리한 쪽이 드랍+정면 동시로 끝냄 | O |
| **역전 드랍** | 0.6~0.8 | 불리한 쪽이 올인 본진 드랍 | X (이후 결전) |

### 게릴라 드랍 내러티브

```
1. 드랍십에 탱크 한 대를 싣고 상대 확장으로 향함
2. 확장기지에 내려서 SCV를 잡음 → 바로 회수
3. 골리앗이 왔지만 이미 빠져나감
4. 드랍으로 시선을 끈 사이 정면 거리재기
```

### 마무리 드랍 내러티브

```
1. 정면에서 앞서고 있음 → 드랍으로 확실하게 끝내려 함
2. 드랍십 두 대 출격 → 본진+확장 동시 투하
3. 정면 탱크 라인도 전진 → 세 방향 공격
4. 수비 분산 → decisive
```

### 역전 드랍 내러티브

```
1. 정면에서 밀리고 있음 → 승부수
2. 드랍십 두 대에 탱크를 싣고 상대 본진으로
3. 본진 탱크 투하 → 시즈 모드 → 팩토리 파괴
4. 피해가 크면 경기 역전 (이후 결전 phase에서 판가름)
```

## 11. 마인(Spider Mine) 규칙

- **호버 유닛(벌처, SCV)은 마인을 밟지 않음**
- 마인은 지상 유닛(탱크, 골리앗, 질럿, 히드라, 울트라 등)만 밟음
- "벌처가 마인에 터진다" 같은 표현 금지
- 벌처는 마인을 **매설**하거나 **피해** 다니는 주체
- 마인이 효과를 발휘하는 시점: 탱크/골리앗 전환 이후

### 올바른 마인 표현

```dart
// O: 벌처가 마인을 심음
'{home} 선수 마인 매설. 탱크 전환 이후를 노리는 포석입니다.'

// O: 탱크가 마인을 밟음
'{away} 선수 탱크가 마인을 밟았습니다! 탱크 한 대가 터집니다!'

// O: 마인 지대 설명
'마인은 벌처는 못 잡지만, 탱크와 골리앗에게는 치명적이죠.'

// X: 벌처가 마인에 피격
'{home} 선수 벌처가 마인에 걸립니다!'  // 금지!
```

## 12. 톤 규칙 (느낌표/마침표)

### 기본 원칙

- **건물/가스/정찰/연구/생산**: 마침표 `.`
- **전투/decisive/견제 성공**: 느낌표 `!`
- 느낌표는 전체 텍스트의 35% 이하로 유지

### 예시

```dart
// O: 건물 건설 → 마침표
'{home} 선수 배럭 건설합니다.'
'{home} 선수 가스를 올립니다.'
'{home} 선수 팩토리 건설을 시작합니다.'

// O: 전투 → 느낌표
'{home} 선수 시즈 포격! 상대 탱크가 터집니다!'
'{home} 선수 벌처 견제 성공! SCV를 잡아냅니다!'

// X: 건물에 느낌표
'{home} 선수 배럭 건설!'  // 과도한 흥분
```

### 선수 이름 뒤 콤마 금지

```dart
// X: 콤마 패턴
'{home}, 배럭 건설합니다.'

// O: '선수' 패턴
'{home} 선수 배럭 건설합니다.'
```

## 13. 종료 시점 분포 가이드

미러전 기준 1000경기 목표 분포:

| 빌드 성격 | 초반 (~28줄) | 중반 (29~42줄) | 후반 (43줄~) |
|-----------|------------|--------------|------------|
| 공격형 (투팩) | 15% | 25~30% | 55~60% |
| 밸런스 (배럭더블) | 15% | 25% | **60%** (가장 높음) |
| 수비형 (팩토리더블) | 15% | 30% | 55% |

- **배럭더블이 가장 장기전** (후반 비율 최고)
- 공격형은 초+중반 합산이 40%+ (빨리 끝나는 경기 많음)
- 수비형은 중반 비율이 높음 (시즈 압도/확장 스킵으로 중반 종료)
- 승률은 미러 기준 45~55%

### 조정 도구

| 분기 | baseProbability ↑ | 효과 |
|------|-------------------|------|
| vulture_crush | 초반 종료 ↑ |
| tank_push | 중반 종료 ↑ |
| siege_crush | 중반~후반 종료 ↑ |
| skip_expand_attack | 중반 종료 ↑ |
| finishing_drop | 후반 종료 중 decisive ↑ |
| desperate_drop | 후반 역전 빈도 ↑ |
